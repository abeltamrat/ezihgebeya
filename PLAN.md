# EzihGebeya Hybrid React Migration Plan

## Ownership legend

Every actionable checklist item below is tagged with who executes it, so tasks can be handed to Codex directly ("do every `[Codex]` item in Phase 1"):

- **[Claude]** — sets a convention, touches money/auth/security, or is a foundation other tasks depend on. Kept with one agent for coherence.
- **[Codex]** — self-contained, independently verifiable, follows a pattern already established. Safe to parallelize.
- **[You]** — not code: hosting/account access, business decisions, pricing, sales/ops, legal content approval.

Untagged items are guardrails, decisions, or acceptance criteria (Scope rules, Execution order, exit gates, the cross-cutting checklists, Definition of done) — not single-owner build tasks.

## Decision

Use a hybrid architecture that preserves the existing public PHP frontend and introduces React where it provides clear value.

```text
Public marketplace and SEO pages
  PHP + HTMX + Alpine.js + DaisyUI/Tailwind

Authenticated application
  /account/*
  /vendor/*
  React + TypeScript + Vite
  (/admin/* stays PHP by default — see Phase 3)

Shared backend
  PHP services + JSON API + MySQL
```

Authentication entry pages remain server-rendered PHP by default:

- `/login`
- `/register`
- `/verify`
- `/forgot-password`

They use the existing PHP session and redirect authenticated users into the appropriate `/app/*` area. They should move to React only if a later review identifies a concrete benefit that justifies rebuilding the OTP and recovery flows.

Canonical route map (one rule, no exceptions elsewhere in this plan):

```text
/login /register /verify /forgot-password   PHP
/ (public pages, browse, detail, business)  PHP
/app/account/*                              React
/app/vendor/*                               React
/admin/*                                    PHP (existing admin retained)
/api/v1/*                                   PHP JSON
```

This direction is chosen because it:

- Preserves the recent Alpine.js, HTMX, DaisyUI 5, and Tailwind 4 work.
- Keeps product, business, category, and content pages search-engine friendly.
- Avoids requiring Node.js, SSR, or a persistent application server on Yegara hosting.
- Uses React for dashboards and state-heavy authenticated workflows.
- Creates APIs that can later support a mobile application.
- Allows useful features to ship incrementally.

## Scope rules

- [ ] Keep public and SEO-sensitive pages rendered by PHP by default.
- [ ] Keep HTMX and Alpine.js for lightweight public-page interactions.
- [ ] Use React for authenticated customer and vendor workflows; keep the existing PHP admin by default (React admin screens only for new capabilities — see Phase 3).
- [ ] Keep login, registration, OTP verification, and password recovery in PHP by default.
- [ ] Build API endpoints only when required by the feature currently being migrated.
- [ ] Migrate one complete feature vertical at a time.
- [ ] Retire each replaced PHP screen after its React version is accepted.
- [ ] Keep authorization, ownership, validation, pricing, and status transitions in PHP.
- [ ] Do not require Node.js in production; deploy only Vite's static build.
- [ ] Reassess the strategy after the first React pilot before expanding it.

## Execution order

The phases below are workstreams, not a strict sequence. The launch-critical order is:

1. **P0 — Production safety**: Phase 1 security work. Nothing ships before this.
2. **P1 — Marketplace fundamentals**: posting quality, attributes, contact-conversion tools, moderation operations, localization — mostly PHP-side work from Phase 4 and the Marketplace fundamentals track. **React must never block this.**
3. **P2 — Liquidity pilot**: recruit real vendors and listings against the liquidity gates before spending on anything else.
4. **P3 — React pilot** (Phase 2), then incremental migration (Phase 3) if the pilot passes.
5. **P4 — Monetization**: only after organic listings receive meaningful inquiries.
6. **P5 — Advanced growth**: push, recommendations, realtime relay, PPC, native wrapper.

A marketplace with 22 listings does not have a technology problem; P1 and P2 are what change that.

### Scale-up migration path

The stack is intentionally boring: PHP 8 + PDO/MySQL + file-based cache + Apache serving `uploads/` and `assets/` directly. Nothing here should change speculatively — each step below is triggered by an observed symptom, not by habit, consistent with this plan's own view that a marketplace with 22 listings does not have a technology problem.

**1. Shared hosting → VPS/cloud host.** No app code changes are required if the existing conventions are respected, because the app already externalizes its environment: `config.php` reads `DB_HOST`/`DB_NAME`/`DB_USER`/`DB_PASS`, `APP_BASE_URL`, `APP_ENV`, `APP_DEV_MODE`, `CRON_SECRET`, and `IP_GEO_API` via `app/env.php`'s `load_env()` from a gitignored `.env` — a VPS just needs its own `.env` with production values, same file format as the current shared-hosting one. `app/db.php`'s `db()` is a single lazy PDO singleton with no host-pooling assumptions. `UPLOAD_DIR` and `CACHE_DIR` are filesystem-absolute directories with no shared-hosting-specific path assumptions. The one manual step is re-adding the two `.htaccess` files' equivalents if the VPS uses nginx instead of Apache: the PHP-execution block in `uploads/.htaccess`, and the root `.htaccess`'s rewrite-to-`index.php` fallback, `Authorization` header passthrough for `/api` bearer tokens, and `Cache-Control` rules. *Trigger:* an actually-observed resource ceiling on the current plan (memory/execution-time caps or CPU throttling under real traffic) — not assumed in advance.

**2. CDN for `assets/`/`uploads/`.** `UPLOAD_URL` (`BASE_URL . '/uploads'`) and every `url('assets/...')` call site flow through single defines/functions rather than being hardcoded per-page, so repointing at a CDN origin is a one-line `config.php` change (e.g. `UPLOAD_URL = getenv('CDN_URL') . '/uploads'`), not a codebase-wide find/replace. Whatever fronts `uploads/` must replicate the PHP-execution deny rule from `uploads/.htaccess` — a CDN doesn't stop someone hitting the origin directly. The `Cache-Control` headers already set in the root `.htaccess` (`max-age=2592000` for long-lived assets/media, `no-cache` for `sw.js`/`manifest.json`) are exactly what a CDN needs to respect for correct edge caching; the service worker's version-aware fetch already handles browser-side staleness, so the CDN just needs an equivalent purge path for `assets/css/app.css`/`assets/js/app.js` on deploy. *Trigger:* uploaded media (product photos/videos) is large enough that origin bandwidth or response time under load becomes the bottleneck — not "CDNs are best practice."

**3. Redis.** Three independent, separately-triggerable plug-in points: (a) `app/cache.php`'s `cache_remember($key, $ttl, $compute)` already isolates all cache I/O behind one function used by every call site (categories, home sections, listing counts) — a Redis-backed version keeps the exact same signature, swapping file I/O for `Redis::get()`/`Redis::set(..., ['EX' => $ttl])`, with zero caller changes; (b) PHP sessions (`index.php` calls plain `session_start()`) can move to Redis via a `session.save_handler`/`session_save_path` ini change alone, no PHP code changes; (c) `rate_limited()` in `app/helpers.php` currently stores attempt timestamps in `$_SESSION`, which only works correctly across multiple app servers if sessions themselves are centralized (see b) — otherwise move its storage to Redis directly (`INCR`+`EXPIRE` per key). *Trigger:* file-cache reads/writes measurably contend (a "cache stampede" under concurrent load), or the app is horizontally scaled across more than one web server, at which point file-based cache and sessions stop being consistent across servers by definition. A single-server deployment has no reason to add Redis at all.

**4. MySQL read replica.** `app/db.php`'s `db()` is a single PDO connection with no read/write split today. Adding a replica means a second connection (e.g. `db_read()`) with a deliberate per-call-site decision: route `pages/browse.php`/`home.php`/`search.php`/`detail.php`/`business.php` (read-only, no same-request write dependency) to the replica; keep `checkout.php`/`vendor_listings.php`/`action_*.php`/`admin.php` and any read-after-write-in-the-same-request (e.g. checkout re-reading the order it just inserted) on the primary to avoid replication-lag showing stale data. `q()`/`row()`/`rows()`/`val()` would need a `$useReplica` parameter or separate `*_read()` wrappers — a deliberate choice per call site, not a blanket switch. *Trigger:* browse/home/search queries show measurable table-scan cost or lock contention with writes (e.g. p95 query time exceeding ~200-300ms under normal load, or `SHOW PROCESSLIST` showing reads queued behind write locks) — and simpler fixes (indexes, tighter `cache_remember()` TTLs on hot paths) have already been tried and are insufficient. A single MySQL instance handles far more read volume than this catalog's current or near-term listing count will produce; a replica is a concurrency/lock-contention fix, not a raw-count fix.

## Revenue model

Research conclusion: commission/take-rate is the global default (reported medians ~9–15% for products, ~20–30% for services) but historically fails in informal, negotiation-driven, cash-preferring markets — transactions close by phone and cash, so the fee is unenforceable and users route around the platform. The survivors in Africa (Jiji) monetize seller visibility, not transactions. Ethiopia adds two specifics: cash-on-delivery still dominates (a ~70% preference is widely reported but not independently validated — treat as hypothesis), and payment gateways (Telebirr, Chapa, SantimPay, ArifPay) are now mature enough to automate the platform's own revenue collection.

### Benchmark: Jiji (the direct competitor, including jiji.com.et)

Jiji is the model to study — and to out-execute — because it operates the visibility model in Ethiopia today:

- Revenue structure: free listings; **TOP ads** (single listing pinned, 7 or 30 days; Kenya reference price ~Ksh 450/week ≈ impulse level — record Ethiopian birr prices before setting ours); **Boost packages** in tiers over 1/3/6/12 months (tier names vary by country and Jiji's own materials are inconsistent — the Ethiopia FAQ documents Basic/Premium/VIP while their 2024 press release names Basic/VIP/Diamond/Enterprise; do not assume a fixed six-tier ladder); **Pro Sales** pay-per-click campaigns with seller-set budgets; packages priced by category group. Note: per Jiji's billing policy their packages are **manually renewed, not auto-renewing subscriptions** — auto-renewal in our plan is our own model choice, not a copy of Jiji.
- Ethiopia scale (self-reported by Jiji at their June 2024 premium-services launch — not independently verified since): **500,000+ monthly users and ~25,000 active sellers**, in an e-commerce market they cited at ~$465M (2024) growing ~13%/year. That is the incumbent we position against — with furniture/services/supplies depth, orders/checkout, and real analytics as the differentiators.
- Distribution: premium packages are sold heavily by an **offline/phone sales team** working weekly targets with a reported ~20% commission on sales — package revenue in this region is sold, not just self-served. Budget for at least one human sales/support role at monetization launch; the "Request help" callback button feeds this.
- Their weaknesses to exploit, stated precisely: Jiji Pro Sales **does** report impressions, clicks, contact/chat conversions, and spend — but their attribution stops at first contact, their organic (non-PPC) seller statistics are thin, and they have no transaction layer at all. Our differentiation is attribution **beyond the contact**: promotion → view → inquiry → quote/order → completion.

### Principles

Guiding principles, not single-owner tasks — every build task in this section must honor them:

- Free listings forever; revenue comes from vendors who are demonstrably earning on the platform, not from entry fees.
- Charge for outcomes vendors can see (visibility, leads, tools), and prove the value with the analytics dashboard — spend vs. attributed inquiries/orders.
- Prioritize ARPU from active vendors over free-user growth; track unit economics per revenue stream from day one.
- Automate the platform's own revenue payments through a gateway; manual payment-proof is acceptable for marketplace orders but kills subscription renewal.

Applied through the real migration runner (not a raw script) so `db_migrations` stays accurate; verified end-to-end — confirmed the index exists via `SHOW INDEX`, confirmed a raw duplicate `INSERT` now throws `SQLSTATE 23000` instead of silently succeeding, confirmed the real HTTP double-submit path correctly shows "You already reviewed this" instead of creating a second row. Two lower-confidence findings flagged but correctly left unfixed by the agent per its scoped instructions (`can_add_listing()`/`can_add_video()`'s plan-limit checks are races without a natural unique-constraint fix — a real but low-severity, self-correcting gap; `find_duplicate_listing()`'s fuzzy title-matching has no exact invariant a constraint could express) — both are legitimate future work, not narrow enough for a same-pass fix.

- **Order/checkout financial consistency**: a dedicated skeptical pass over money specifically, since it hadn't had one yet despite being the most consequential area if wrong. Found and reproduced a serious, real bug: `pages/account_orders.php`'s "pay" handler had no server-side guard against submitting a second payment against an order that already had one pending/confirmed — the button was only hidden client-side. The agent proved this concretely: bypassed the hidden button with a direct POST, got two separate `pending` payment rows against one order, then confirmed both via the real admin flow (the existing `WHERE status='pending'` guard only stops re-confirming the *same* row, not a *second, separate* duplicate row) — landing `SUM(payments.amount) = 141,400 ETB` confirmed against a `70,700 ETB` order, a genuinely corrupted payments ledger that would flow into any GMV/payout report built on that sum. Fixed both: `account_orders.php` now checks for an existing pending/confirmed payment before inserting a new one; `admin.php`'s `payment_confirm`/`payment_reject` now performs one atomic `UPDATE ... WHERE status = 'pending'` and only proceeds with the linked order/promotion/subscription activation if that specific request's UPDATE actually affected a row, closing the confirm-side race too. Verified end-to-end against the real dev DB and the exact reproduced scenario: attempted a duplicate payment submission against an order that already had a confirmed payment — correctly blocked with "A payment for this order is already submitted or confirmed," payment count stayed at 1, confirmed sum stayed at 70,700 ETB; fired two near-simultaneous confirm requests against a fresh pending payment — exactly one notification was created, confirming the activation path ran only once. Two related, lower-severity items surfaced by the same audit and explicitly not fixed (deliberately out of this pass's narrow scope): admin's `order_status` action has no guard preventing an order with a confirmed payment from being cancelled with no refund record created (a manual-process gap, not silent corruption); and the admin Orders table doesn't show payment status inline, requiring a cross-reference to the Payments tab (a minor UX gap). `orders.total` vs. `order_items` was confirmed structurally sound — line items are only ever inserted once at checkout, no edit path exists that could cause divergence.

**Fifth round of parallel audits** (3 more agents): checkout price/stock consistency, premium-downgrade revocation, and a dedicated ownership pass over the bearer-token API (`pages/api.php`, distinct from `/api/v1`'s already-audited React API).

- **Bearer-token API ownership audit**: `pages/api.php` (the legacy non-browser REST API, separate from the already-clean `/api/v1`) confirmed sound on authentication (real SHA-256 token lookup against an active user, not "any header accepted") and ownership scoping (every query filters by the token's own `customer_id`/`user_id`, never a client-supplied id) — and its attack surface for the IDOR bug class is narrower by design anyway, since it doesn't expose vendor-mutation endpoints at all (no listing/business edit, no orders/payments). **Sixth round of parallel audits** (3 more agents), pivoting away from "same bug class elsewhere" toward genuinely distinct security angles, since the previous round showed early signs of diminishing returns on that pattern:

- **Session fixation**: audited whether login, registration, password reset, and logout correctly regenerate the session ID at privilege boundaries. Login, registration, and account deletion all confirmed correct (verified live: session ID visibly changes across login). Found and fixed a real gap: `pages/auth_logout.php` called only `session_destroy()`, which clears server-side data but does **not** clear `$_SESSION`, expire the client's cookie, or regenerate the ID — verified live *before* the fix that logout's response carried no `Set-Cookie` header at all, meaning a browser would keep resending the exact same session ID after "logging out." Fixed to the full sequence (`session_unset()` → explicit cookie expiry via `setcookie()` → `session_destroy()` → `session_regenerate_id(true)`), matching the pattern already used elsewhere in the codebase (idle timeout, account deletion) plus the cookie-clear step those call sites didn't need. Re-verified independently: logout now returns a correct `Set-Cookie: PHPSESSID=deleted; expires=1970...; Max-Age=0` and the next request gets a fresh session ID. Password reset doesn't regenerate the session ID either and doesn't invalidate other active sessions on the account — flagged as a real but lower-severity gap requiring a broader "invalidate all sessions on credential change" feature (would need a sessions table or per-user token version) rather than a one-line fix, left for a deliberate decision. Confirmed no code path anywhere accepts a session ID from a URL parameter or any non-cookie source, so classic link-based fixation vectors don't apply regardless.
- **Upload security (deep pass)**: found and fixed the most severe bug of this entire session — **real, demonstrated stored XSS** via AR model uploads. `upload_model()` (unlike `upload_image()`) did zero content validation, trusting only the client-supplied filename extension. Proved this concretely: renamed a raw HTML file containing `<script>alert(document.cookie)</script>` to `xsstest.glb`, uploaded it, and confirmed via direct HTTP fetch that Apache has no MIME mapping for `.glb`/`.usdz`, sends **no `Content-Type` header at all**, and served the file's raw HTML/script content with a `200` — meaning a browser opening that URL directly would MIME-sniff and execute it. Reachable by any vendor on a premium-plan business (`feature_enabled('ar')` + `current_plan() === 'premium'`), a real, live vendor-facing upload path, not a theoretical one. Fixed with magic-byte validation in `upload_model()`: `.glb` files must start with the glTF binary signature, `.usdz` (a zip container) must start with a ZIP local-file-header signature — rejecting anything else regardless of claimed extension. Also added `X-Content-Type-Options: nosniff` to `uploads/.htaccess` site-wide as defense-in-depth, so even an unanticipated future extension/content mismatch can't be browser-sniffed into executing as HTML. Independently re-verified via a fresh real HTTP multipart upload as a real premium vendor: the malicious file is now rejected — confirmed via database ground truth that no `product_media` row was created and no file was written to `uploads/ar-models/` — while the underlying magic-byte check itself was separately confirmed to correctly accept a genuine glTF header (a direct-function test hit an unrelated `move_uploaded_file()` limitation specific to CLI-script testing, not a flaw in the fix). SVG uploads confirmed already safe (not in `upload_image()`'s MIME allowlist at all). Also found and fixed a real, narrower gap: the admin uploads-restore tool (`admin_more_actions.php`, super-admin only) had no zip-bomb protection — a tiny crafted archive claiming a huge uncompressed size could exhaust disk during extraction. Fixed with pre-extraction entry-count/per-entry-size/total-size limits checked via `statIndex()` before any file is written. Lower severity given super-admin-only reach, but fixed alongside the higher-severity finding since it was found in the same pass.
- **User-content escaping (notifications/email/SMS/admin)**: confirmed clean across every surface checked — in-app notifications store raw and escape correctly at render (`e()` in `pages/notifications.php`), email/SMS are plain-text with no HTML-injection surface and no header-injection vector (the `From` header is admin-config only, never user input; the `To` address is validated via `FILTER_VALIDATE_EMAIL` at registration, which rejects the CRLF sequences a header-injection attack needs), the admin panel escapes every user-submitted field checked (reviews, reports, support tickets, business/user names, inquiries) with the same discipline as public pages, and flash messages — even ones that echo raw user input — are safe by construction regardless of content, since they flow through `json_encode()` (escapes `/`, blocking script-tag breakout) into an Alpine `x-text` binding (sets `textContent`, never `innerHTML`, so embedded markup can never execute even unescaped). No gap found, no changes made.

**Seventh round of parallel audits** (3 more agents): admin vs. super_admin privilege boundary, self-dealing/metric-gaming, and cryptographic hygiene.

- **Admin/super_admin boundary**: audited every `$do === '...'` branch in `pages/admin.php` and `admin_more_actions.php` against what the UI hides from regular admins (backups, migrations, system-UI theming, ad manager, admin account management). Found and fixed a real, deceptive gap: `pages/action_review.php` had zero self-review guard — only a duplicate-review check, not an ownership check — so any vendor could 5-star their own business. This directly matters because `businesses.rating_average` feeds both the §8.4 ranking-score `rating` weight and the video-feed ranking's `rating` weight, and is shown publicly (star ratings on business/detail pages, plus `aggregateRating`/`ratingValue` in JSON-LD structured data search engines read) — a real, external, deceptive effect, not cosmetic. Independently re-verified end-to-end: confirmed the real owner of a real business is now blocked from reviewing it (0 rows created), and confirmed a genuinely different customer can still review the same business normally afterward (no regression). Checked four related but lower-stakes vectors and correctly left them alone: favorites and inquiries self-dealing have no external deceptive effect since neither feeds the default ranking score or is shown to buyers; view-count self-inflation is an accepted industry-wide tradeoff and also isn't in the default ranking formula; video engagement self-inflation is already log-dampened and capped, bounded by design.
- **Cryptographic hygiene**: a foundational check of the actual primitives, not just "is a check present" — came back completely clean. CSRF tokens use `random_bytes(20)` (160 bits) compared via `hash_equals()`; OTP codes use `random_int()` (not `rand()`/`mt_rand()`) with a 5-attempt lockout and prior-code invalidation; every password-write path — including the admin account-creation shortcut, historically a common place for such shortcuts to diverge — uses `password_hash(..., PASSWORD_BCRYPT)` with `password_verify()` on every read path, no `md5`/`sha1`/plaintext found anywhere; every `session_regenerate_id()` call site (including all the ones fixed earlier this session) correctly passes `(true)`; bearer API tokens are minted via `random_bytes(32)` and only ever persisted as a SHA-256 hash, never plaintext. The one `mt_rand()` found in the codebase (`app/ads.php`, weighted ad-rotation lottery) is correctly non-security-sensitive and out of scope. No gap found, no changes made.
